# cal-work MVP — Final Architecture Specification

## 1. Overview

**Goal:** Allow a host to create meeting types, share a link, and let guests book a slot.

**Scope:** One hardcoded user, no authentication, no email delivery, no Google Calendar sync.

```
Host creates EventType → gets link → sends to guest
Guest opens link → picks a free slot → fills name/email → booking confirmed
```

## 4. Database Schema (Drizzle / SQLite)

| -------------- | ------------------------------------- | --------------- |

| `display_name` | TEXT NOT NULL                         | "Alex Sokolov"  |
| `timezone`     | TEXT NOT NULL DEFAULT 'Europe/Moscow' | IANA identifier |

### `event_types`

| Column        | Type                     | Notes                    |
| ------------- | ------------------------ | ------------------------ |
| `id`          | INTEGER PK AUTO          |                          |
| `user_id`     | TEXT FK → users.username |                          |
| `title`       | TEXT NOT NULL            | "30 Minute Consultation" |
| `slug`        | TEXT NOT NULL            | "30min", unique per user |
| `description` | TEXT NULLABLE            |                          |
| `duration`    | INTEGER NOT NULL         | Minutes: 15, 30, 45, 60  |

Unique constraint: `(user_id, slug)`

### `availability`

| Column    | Type                            | Notes                  |
| --------- | ------------------------------- | ---------------------- |
| `id`      | INTEGER PK AUTO                 |                        |
| `user_id` | TEXT UNIQUE FK → users.username | One row per user       |
| `slots`   | TEXT NOT NULL                   | JSON array (see below) |

**`slots` JSON format:**

```json
[
  { "dayOfWeek": 1, "start": "09:00", "end": "17:00" },
  { "dayOfWeek": 3, "start": "10:00", "end": "15:00" }
]
```

- `dayOfWeek`: 1=Monday … 7=Sunday
- `start`/`end`: HH:mm in 24h format, in the host's timezone

### `bookings`

| Column          | Type                        | Notes        |
| --------------- | --------------------------- | ------------ |
| `id`            | INTEGER PK AUTO             |              |
| `event_type_id` | INTEGER FK → event_types.id |              |
| `guest_name`    | TEXT NOT NULL               |              |
| `guest_email`   | TEXT NOT NULL               |              |
| `start_time`    | TEXT NOT NULL               | ISO 8601 UTC |
| `end_time`      | TEXT NOT NULL               | ISO 8601 UTC |
| `created_at`    | TEXT NOT NULL               | ISO 8601 UTC |

| -------- | ----------------------- | --------------------------------- |
| `GET`    | `/api/event-types`      | List event types for current user |
| `POST`   | `/api/event-types`      | Create a new event type           |
| `GET`    | `/api/event-types/{id}` | Get single event type             |
| `PATCH`  | `/api/event-types/{id}` | Update event type                 |
| `DELETE` | `/api/event-types/{id}` | Delete event type                 |

### Availability

| Method | Path                | Description                     |
| ------ | ------------------- | ------------------------------- |
| `GET`  | `/api/availability` | Get current user's availability |
| `PUT`  | `/api/availability` | Create or replace availability  |

### Booking (public — no auth)

| Method | Path                                                | Description     |
| ------ | --------------------------------------------------- | --------------- |
| `GET`  | `/api/book/{username}/{slug}/slots?from=ISO&to=ISO` | List free slots |
| `POST` | `/api/book/{username}/{slug}`                       | Create booking  |

#### `GET /api/book/{username}/{slug}/slots`

Query params:

- `from` — ISO 8601 UTC start (required)
- `to` — ISO 8601 UTC end (required)

Response:

```json
{
  "eventType": {
    "id": 1,
    "title": "30 Minute Consultation",
    "duration": 30
  },
  "timezone": "Europe/Moscow",
  "slots": [
    { "start": "2026-07-06T06:00:00.000Z", "end": "2026-07-06T06:30:00.000Z" },
    { "start": "2026-07-06T06:30:00.000Z", "end": "2026-07-06T07:00:00.000Z" }
  ]
}
```

#### `POST /api/book/{username}/{slug}`

Body:

```json
{
  "startTime": "2026-07-06T06:00:00.000Z",
  "guestName": "Ivan Petrov",
  "guestEmail": "ivan@example.com"
}
```

Response: `201` with booking details + redirect to confirmed page.

## 6. Slot Calculation Algorithm

### `slotService.ts` — `computeFreeSlots(params): Slot[]`

```
Input:
  - availability: { dayOfWeek, start, end }[]
  - eventTypeDuration: minutes (30)
  - existingBookings: { startTime, endTime }[]
  - timezone: IANA string ("Europe/Moscow")
  - from: DateTime (UTC)
  - to: DateTime (UTC)
  - minimumNotice: minutes (240 = 4 hours default)

Algorithm:
  1. cutoffTime = now UTC + minimumNotice
  2. For each calendar day in [from, to]:
     a. Get dayOfWeek (1-7) from the date in host's timezone
     b. Find availability windows for that dayOfWeek
     c. For each window:
        - Convert window start/end to UTC DateTime for this specific date
          using the host's timezone (handles DST correctly)
        - Generate slots of eventTypeDuration minutes:
          slotStart = windowStart, slotEnd = slotStart + duration
          while slotEnd <= windowEnd
        - Filter out:
          * Slots where slotStart < cutoffTime
          * Slots that overlap with any existing booking
            (overlap when: slotStart < bookingEnd AND slotEnd > bookingStart)
     d. Add valid slots to result
  3. Sort result by start time, return
```

### Overlap detection

```
Two intervals [A_start, A_end) and [B_start, B_end) overlap when:
  A_start < B_end AND A_end > B_start
```

## 7. Frontend — Pages & Components

### Routes (React Router v7)

| Path                                  | Page                  | Purpose                                  |
| ------------------------------------- | --------------------- | ---------------------------------------- |
| `/dashboard`                          | `DashboardPage`       | List event types + availability settings |
| `/dashboard/event-types/new`          | `EventTypeCreatePage` | Create form                              |
| `/book/:username/:slug`               | `BookingPage`         | Guest booking flow                       |
| `/book/:username/:slug/confirmed/:id` | `ConfirmedPage`       | Booking confirmation                     |

Any unknown path → redirect to `/dashboard`.

### Page Details

#### [`DashboardPage`](web/src/pages/DashboardPage.tsx)

- **Data:** `GET /api/event-types`, `GET /api/availability`
- **Layout:**
  - Header: "cal-work" logo + username
  - Section: "Your Event Types" — grid of `EventTypeCard` components + "Create" button
  - Section: "Availability" — `AvailabilityEditor` component (expandable/collapsible)
- **EventTypeCard:** title, duration badge, "Copy link" button → copies booking URL
- **AvailabilityEditor:** checkboxes for days Mon-Sun, time range inputs for each active day

#### [`EventTypeCreatePage`](web/src/pages/EventTypeCreatePage.tsx)

- Title input → auto-generates slug
- Slug input (editable, validated: `^[a-z0-9-]+$`)
- Duration dropdown: 15/30/45/60
- Description textarea (optional)
- Submit → `POST /api/event-types` → redirect `/dashboard`
- Slug conflict → `alert()` with error

#### [`BookingPage`](web/src/pages/BookingPage.tsx)

Two-step internal state (not separate routes):

**Step 1 — Slot Selection:**

- Fetches `GET /api/book/{username}/{slug}/slots?from=now&to=now+7days`
- Header: event type title, duration, host display name
- Date picker: horizontal scroll of next 7 days, highlights days with slots
- Selected day shows time slots grouped by morning/afternoon/evening
- All times shown in host's timezone
- Click slot → transition to step 2

**Step 2 — Booking Form:**

- Shows selected: "Mon 6 July, 10:00 - 10:30 MSK"
- Fields: full name, email (HTML `required` only, server validates)
- Submit → `POST /api/book/{username}/{slug}` → redirect to confirmed page
- Errors → `alert()`

#### [`ConfirmedPage`](web/src/pages/ConfirmedPage.tsx)

- Fetches `GET /api/bookings/{bookingId}` — survives page refresh
- Green checkmark, "You're booked!"
- Shows: event title, date/time, timezone

## 9. Dev Workflow — Frontend-First with Prism Mocks

**Strategy:** Build the React frontend first, using Prism to mock the API.
Real Express backend comes AFTER the frontend works end-to-end against mocks.

### Prism — Mock Server from OpenAPI

Prism reads [`openapi.yaml`](openapi.yaml) and starts a fake API server that:

- Serves **realistic mock responses** (valid JSON matching schemas)
- **Validates requests** against the spec (wrong body → 400)
- Supports **dynamic IDs** in paths (`/event-types/42` works)
- Generates data with correct types (`safeint` → integer, `utcDateTime` → ISO date)

```bash
make mock   # → Prism on :4010
```

### Three-Terminal Setup

```bash
# Terminal 1: TypeSpec watch mode
make spec-watch

# Terminal 2: Prism mock server
make mock

# Terminal 3: Frontend dev server
make dev   # Vite on :5173, proxy /api → Prism :4010
```

### Swagger UI (Auto-Updating)

Open `http://localhost:5173/swagger.html` after `make dev`.

**Why it auto-updates:** `web/public/openapi.yaml` is a **symlink** to the generated spec in the project root. After `tsp compile`, Vite sees the change through the symlink instantly — no copy needed.

### Vite Proxy Config

```typescript
// web/vite.config.ts
export default defineConfig({
  server: {
    proxy: {
      "/api": {
        target: "http://localhost:4010", // Prism mock in dev; switch to :3000 for real backend
        changeOrigin: true,
      },
    },
  },
});
```

### Workflow: Making an API Change

```
1. Edit main.tsp                    → add new field or endpoint
2. make spec                        → auto-regenerates openapi.yaml (symlink updated)
3. Prism auto-reloads               → serves updated mock responses
4. make types                       → regenerate TS types
5. Refresh browser                  → Swagger shows new endpoints, frontend sees new types
```

### When to Switch to Real Backend

After all 4 frontend pages work correctly against Prism mocks, implement the real Express backend. The frontend code **does not change** — just point Vite proxy from Prism (`:4010`) to Express (`:3000`).

## 10. Verification & Testing Strategy

### Layer 1 — TypeSpec Compilation (contract integrity)

```bash
make spec   # Fail = broken API contract
```

Runs on every change. Ensures models and routes are self-consistent.

### Layer 2 — API Contract Compliance (runtime validation)

Using `express-openapi-validator` middleware on the Express backend:

```typescript
import * as OpenApiValidator from "express-openapi-validator";

app.use(
  OpenApiValidator.middleware({
    apiSpec: "./openapi.yaml",
    validateRequests: true, // Bad request → 400 automatically
    validateResponses: true, // Wrong response shape → warning
  }),
);
```

Every request and response is validated against the OpenAPI spec at runtime. No manual checks needed.

### Layer 3 — Unit Tests: Slot Calculation Algorithm (vitest)

```typescript
// server/src/services/__tests__/slotService.test.ts
describe("computeFreeSlots", () => {
  it("generates slots from a single availability window");
  it("excludes slots that overlap with existing bookings");
  it("skips slots before the minimum notice cutoff");
  it("handles DST transition correctly");
  it("returns empty array when no availability for the day");
  it("trims partial slots at window boundaries");
  it("handles edge case: slot exactly equals window");
});
```

**Critical:** The algorithm is a pure function `(inputs) → (slots)`. No database in these tests — only mocked data. Fast, deterministic, easy to reason about.

### Layer 4 — Integration Tests: API Routes (vitest + supertest + SQLite :memory:)

```typescript
// server/src/routes/__tests__/booking.test.ts
describe("GET /api/book/:username/:slug/slots", () => {
  it("returns 404 for unknown username");
  it("returns 404 for unknown slug");
  it("returns free slots for a valid event type with availability");
  it("excludes slots already booked");
});

describe("POST /api/book/:username/:slug", () => {
  it("creates a booking and returns 201");
  it("returns 409 when slot is already taken (double booking)");
  it("returns 400 when slot is outside availability");
});
```

SQLite `:memory:` mode gives each test a fresh database. Migrations + seed run before tests.

### Layer 5 — Frontend Build Check

```bash
cd web && npm run build   # TypeScript + Vite build, must pass with 0 errors
```

### Layer 6 — Manual Smoke Test (MVP acceptance)

Critical path walkthrough:

1. Open dashboard → create event type → copy booking link
2. Open link in incognito → see free slots → pick one → fill form → confirm
3. Open link again → the booked slot is no longer available

### CI Pipeline (when GitHub is set up)

```yaml
steps:
  - run: make spec
  - run: cd server && npm test
  - run: cd server && npm run typecheck
  - run: cd web && npm run build
```

## 11. What is NOT in MVP

- Authentication / multi-user
- Email notifications
- Google Calendar / Outlook sync
- Booking cancellation / reschedule by guest
- Video conferencing links (Zoom, Meet)
- Custom booking questions
- Buffer time between meetings
- Date-specific availability overrides
- Admin panel
